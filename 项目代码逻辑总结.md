# 尚硅谷AI小智项目 — 代码逻辑总结

---

## 一、项目总览

### 1.1 项目简介

本项目基于 ESP32-S3 平台，实现了一个 AI 语音助手（"小智"），核心功能包括：

- **语音唤醒**：通过 WakeNet 模型检测唤醒词
- **语音活动检测（VAD）**：判断用户是否在说话
- **音频编解码**：PCM ↔ Opus 压缩/解压
- **网络通信**：BLE 配网 → HTTP 激活 → WebSocket 语音交互
- **显示交互**：LVGL 驱动 LCD 屏幕，显示表情、文字、二维码
- **按键交互**：ADC 按键支持单击/长按

### 1.2 整体数据流

```
麦克风 → I2S → ES8311 AFE处理(降噪/VAD/唤醒) → RingBuffer → Opus编码 → RingBuffer → WebSocket
                                                                                        ↑
扬声器 ← I2S ← ES8311 ← Opus解码 ← RingBuffer ← WebSocket
```

### 1.3 任务架构（FreeRTOS 多任务双核）

| 任务名 | 绑定核心 | 优先级 | 功能 |
|--------|---------|--------|------|
| `sr_feed` | CPU0 | 5 | 麦克风音频采集，喂入 AFE |
| `sr_get` | CPU1 | 5 | AFE 结果处理，唤醒/VAD 检测 |
| `opus_encoder_task` | CPU1 | 5 | PCM → Opus 编码 |
| `opus_decoder_task` | CPU0 | 5 | Opus → PCM 解码 + 播放 |
| `encoder_decoder_test_task` | 默认 | 5 | 测试用：编码输出 → 解码输入（正式版由 WebSocket 替代） |
| `lvgl` | 无亲和 | 4 | LVGL UI 渲染 |

### 1.4 RingBuffer 拓扑

| 名称 | 大小 | 类型 | 生产者 | 消费者 |
|------|------|------|--------|--------|
| `sr2encoder_ringbuffer` | 8KB | `RINGBUF_TYPE_BYTEBUF` | sr_get（AFE输出的PCM） | opus_encoder_task |
| `encoder2ws_ringbuffer` | 16KB | `RINGBUF_TYPE_NOSPLIT` | opus_encoder_task | WebSocket（当前为测试任务） |
| `decoder_in_ring` | 16KB | `RINGBUF_TYPE_NOSPLIT` | WebSocket（当前为测试任务） | opus_decoder_task |

## 二、模块详解

### 2.1 音频硬件编解码 — `xiaozhi_hard_codec`

**文件**：`main/audio/xiaozhi_hard_codec.c/.h`

#### 功能

初始化 ES8311 音频编解码芯片，提供音频播放和录制接口。

#### 硬件接口

| 接口 | 引脚 | 用途 |
|------|------|------|
| I2C SCL | GPIO1 | ES8311 寄存器控制 |
| I2C SDA | GPIO0 | ES8311 寄存器控制 |
| I2S BCK | GPIO2 | 音频位时钟 |
| I2S MCK | GPIO3 | 音频主时钟 |
| I2S WS | GPIO5 | 音频字选择 |
| I2S DOUT | GPIO6 | 音频数据输出（播放） |
| I2S DIN | GPIO4 | 音频数据输入（录音） |
| PA | GPIO7 | 功放控制 |

#### 初始化流程

```
1. I2C 总线初始化 (i2c_new_master_bus)
2. I2S 通道初始化 (i2s_new_channel → i2s_channel_init_std_mode)
3. 数据接口创建 (audio_codec_new_i2s_data) → data_if
4. 控制接口创建 (audio_codec_new_i2c_ctrl) → ctrl_if
5. GPIO 接口创建 (audio_codec_new_gpio) → gpio_if
6. ES8311 编解码器创建 (es8311_codec_new) → codec_if
7. 设备句柄创建 (esp_codec_dev_new) → codec_dev
8. 配置采样率 16kHz / 单声道 / 16bit，打开设备
9. 设置输出音量 50%，输入增益 20
```

#### 核心API

| 函数 | 功能 |
|------|------|
| `xiaozhi_codec_init()` | 初始化整个音频硬件链路 |
| `xiaozhi_codec_play(data, len)` | 播放PCM音频数据 |
| `xiaozhi_codec_record(data, size)` | 录制PCM音频数据 |

#### 硬件抽象设计

```c
es8311_codec_cfg_t es8311_cfg = {
    .codec_mode = ESP_CODEC_DEV_WORK_MODE_BOTH,
    .ctrl_if    = out_ctrl_if,   // I2C控制接口
    .gpio_if    = gpio_if,       // GPIO接口
    .pa_pin     = GPIO_NUM_7,
    .use_mclk   = true,
};
out_codec_if = es8311_codec_new(&es8311_cfg);
```

这种分层设计使得更换音频芯片（如从 ES8311 换成 ES8388）时，只需修改 `es8311_codec_new` 为 `es8388_codec_new`，上层代码（`esp_codec_dev_write/read`）无需改动。

### 2.2 语音识别前端 — `xiaozhi_sr`

**文件**：`main/audio/xiaozhi_sr.c/.h`

#### 功能

使用 ESP AFE（Audio Front-End）框架实现：
- **唤醒词检测**（WakeNet）
- **语音活动检测**（VAD）
- **音频前处理**（AEC 回声消除等，当前已关闭）

#### AFE 配置参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `afe_type` | `AFE_TYPE_SR` | 语音识别模式 |
| `afe_mode` | `AFE_MODE_HIGH_PERF` | 高性能模式 |
| `vad_min_noise_ms` | 1000 | 静音持续1s才判定语音结束 |
| `vad_min_speech_ms` | 300 | 最短语音300ms，避免噪声误触 |
| `vad_mode` | `VAD_MODE_1` | 语音触发概率适中 |
| `aec_init` | false | 关闭回声消除 |
| `se_init` | false | 关闭语音增强 |

#### 任务1：sr_feed（CPU0）

```
循环：
  1. 从I2S读取麦克风数据 → audio_buffer
  2. 喂入AFE → afe_handle->feed(afe_data, audio_buffer)
```

实时性要求高，独占 CPU0。

#### 任务2：sr_get（CPU1）

```
循环：
  1. 从AFE取处理结果 → afe_handle->fetch(afe_data)
  2. 检查唤醒状态：
     - WAKENET_DETECTED → 设置 wake_flag=true，执行唤醒回调
  3. 唤醒后跟踪VAD状态变化：
     - SPEECH → SILENCE：语音结束，清除 wake_flag
     - 状态变化时调用 vad_change_callback
  4. 如果 wake_flag && VAD_SPEECH：
     - 将处理后的PCM数据写入 sr2encoder_ringbuffer
     - 如果有 vad_cache（截断音频），拼接到数据前面
```

#### VAD 状态机

```
未唤醒 ──[唤醒词]──→ 唤醒状态(wake_flag=true, VAD=SILENCE)
                         │
                    [检测到人声]
                         ↓
                    说话中(wake_flag=true, VAD=SPEECH)
                    → 音频送入编码器
                         │
                    [静音1秒]
                         ↓
                    交互结束(wake_flag=false)
```

### 2.3 Opus 编码器 — `xiaozhi_opus_encodec`

**文件**：`main/audio/xiaozhi_opus_encodec.c/.h`

#### 什么是 Opus？

Opus 是一个开源、高效的音频编码格式，专为实时语音和音乐传输设计。在 VoIP 场景下，能将 16kHz/16bit/单声道 PCM 数据压缩约 **10倍**，极大降低网络带宽需求。

#### 编码参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `sample_rate` | 16kHz | 采样率 |
| `channel` | 单声道 | 麦克风输入 |
| `bits_per_sample` | 16bit | 采样位深 |
| `bitrate` | 20kbps | 编码输出码率 |
| `frame_duration` | 60ms | 每帧时长 |
| `application_mode` | VOIP | 语音优化模式 |

#### 帧大小计算

```zhuti
PCM帧大小 = 采样率 × 时长 × 通道数 × 位深/8
          = 16000 × 0.06 × 1 × 2
          = 1920 字节

Opus帧大小 ≈ zhu'ti × 时长 / 8
           = 20000 × 0.06 / 8
           ≈ 150 字节
```

压缩比 ≈ 1920:150 ≈ **12.8:1**

#### 编码任务流程

```
循环：
  1. 从 sr2encoder_ringbuffer 累积读取 1920 字节 PCM
     （可能需要多次读取，因为 RingBuffer 中数据可能不足一帧）
  2. 调用 esp_opus_enc_process 编码
  3. 将编码后的 Opus 数据（约150字节）写入 encoder2ws_ringbuffer
```

#### 为什么用 RingBuffer？

- **生产者-消费者解耦**：sr_get 产出 PCM 是连续流，编码器消费是固定帧大小（1920字节），两者速率不同
- **跨任务安全**：FreeRTOS RingBuffer 自带线程安全，无需额外锁
- **背压控制**：RingBuffer 满时生产者阻塞，避免内存溢出

### 2.4 Opus 解码器 — `xiaozhi_opus_decoder`

**文件**：`main/audio/xiaozhi_opus_decoder.c/.h`

#### 解码参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `sample_rate` | 16kHz | 与编码器一致 |
| `channel` | 单声道 | 与编码器一致 |
| `frame_duration` | 60ms | 与编码器一致 |
| `self_delimited` | false | 不带自分隔头 |

#### 解码任务流程

```
循环：
  1. 从 decoder_in_ring 读取一整帧 Opus 数据
  2. 调用 esp_opus_dec_decode 解码为 PCM
  3. 调用 xiaozhi_codec_play 播放 PCM 数据
  4. 处理 consumed 偏移，继续解码剩余数据
  5. 释放 RingBuffer 数据项
```

#### RingBuffer 类型选择

| RingBuffer | 类型 | 原因 |
|-----------|------|------|
| `sr2encoder` | `BYTEBUF` | PCM 是连续字节流，无需按帧边界对齐 |
| `encoder2ws` | `NOSPLIT` | Opus 帧是独立完整单元，读取时必须拿到完整一帧 |
| `decoder_in` | `NOSPLIT` | 同上，解码器需要完整 Opus 帧才能解码 |

`NOSPLIT` 保证写入的每条数据在读取时不会被截断，适合帧结构的消息传递。

### 2.6 按键模块 — `xiaozhi_button`

**文件**：`main/button/xiaozhi_button.c/.h`

#### 功能

通过 ADC 电压区分实现两个物理按键（共用同一个 ADC 通道7）。

#### 按键配置

| 按键 | ADC范围 | 位置 | 事件 |
|------|---------|------|------|
| SW2 | 0~500mV（接近GND） | 前面板 | 单击 → 未激活时重启设备 |
| SW3 | 1000~2400mV | 后面板 | 单击 / 长按 → 长按重置WiFi |

#### ADC 按键原理

多个按键通过不同阻值的分压电路连接到同一个 ADC 通道，不同按键按下时产生不同的电压值，软件根据电压范围判断是哪个按键。

#### 按键事件回调

```c
void button_cb(void *button_handle, void *user_data) {
    uint8_t event = (uint8_t)user_data;
    if (event == 1) { /* 前键单击：未激活→重启 */ }
    if (event == 2) { /* 后键单击 */ }
    if (event == 3) { /* 后键长按：重置WiFi */ }
}
```

### 2.7 WiFi 配网 — `xiaozhi_wifi`

**文件**：`main/internet/xiaozhi_wifi.c/.h`

#### 功能

通过 BLE（蓝牙低功耗）实现 WiFi 配网，用户使用手机 APP 扫描二维码或蓝牙搜索设备，输入 WiFi 密码。

#### 配网流程

```
1. NVS 初始化
2. 创建网络接口和事件循环
3. 注册事件回调（WiFi/IP/配网/BLE/安全会话）
4. 检查是否已配网（NVS中有记录）
   ├── 已配网 → 直接以 STA 模式连接WiFi
   └── 未配网 → 启动BLE配网服务
       ├── 生成设备名 PROV_XXXXXX（MAC后3字节）
       ├── 安全方案 SECURITY_1（对称加密，PoP=abcd1234）
       ├── 屏幕显示配网二维码
       └── 等待手机APP通过BLE传入WiFi凭据
5. 配网成功 → WiFi连接 → 获取IP
```

#### 事件处理

| 事件 | 处理 |
|------|------|
| `WIFI_EVENT_STA_START` | 发起WiFi连接 |
| `WIFI_EVENT_STA_DISCONNECTED` | 自动重连 |
| `IP_EVENT_STA_GOT_IP` | 设置连接成功标志 |
| `WIFI_PROV_CRED_RECV` | 日志记录收到的WiFi凭据 |
| `WIFI_PROV_CRED_FAIL` | 重置配网管理器状态 |
| `WIFI_PROV_CRED_SUCCESS` | 配网成功 |
| `WIFI_PROV_END` | 释放配网资源 |

#### WiFi 重置

`xiaozhi_wifi_reset()`：恢复 WiFi 状态并重启设备，用于重新配网。

### 2.8 设备激活 — `xiaozhi_active`

**文件**：`main/internet/xiaozhi_active.c/.h`

#### 功能

通过 HTTPS POST 请求向服务器注册设备，获取 WebSocket 连接信息和激活状态。

#### 激活请求

- **URL**：`https://api.tenclass.net/xiaozhi/ota/`
- **方法**：POST（HTTPS/TLS）
- **请求头**：

| Header | 值 | 说明 |
|--------|-----|------|
| `Content-Type` | `application/json` | JSON格式 |
| `Device-Id` | MAC地址 | 设备唯一标识 |
| `Client-Id` | UUID v4 | 客户端唯一标识（NVS持久化） |

- **请求体**：
```json
{
  "application": {
    "version": "1.0.1",
    "elf_sha256": "<固件SHA256>"
  },
  "board": {
    "type": "bread-compact-wifi",
    "name": "bread-compact-wifi-128x64",
    "ssid": "...",
    "rssi": -55,
    "mac": "xx:xx:xx:xx:xx:xx"
  }
}
```

#### 响应解析

| 字段 | 说明 |
|------|------|
| `websocket.url` | WebSocket 服务地址（持久化到 xiaozhi_handle） |
| `websocket.token` | WebSocket 认证令牌 |
| `activation` | 存在 → 未激活，包含 `code`（6位激活码） |
| `activation` | 不存在 → 已激活 |

#### UUID v4 生成

使用 `esp_random()` 生成随机数，按 UUID v4 规范设置版本号和变体位，首次生成后存入 NVS，后续复用。

## 五、关键设计决策

### 5.1 双核任务绑定

- **CPU0**：音频采集（sr_feed）、音频播放（decoder）— 实时性要求高
- **CPU1**：AFE处理（sr_get）、Opus编码 — 计算密集型

### 5.2 PSRAM 分配

所有大缓冲区（RingBuffer、编解码缓冲区、任务栈）均分配到 PSRAM，节省内部 SRAM。

### 5.3 接口抽象

音频硬件通过 `audio_codec_if_t` / `audio_codec_ctrl_if_t` / `audio_codec_data_if_t` 三层接口抽象，实现硬件无关的编解码操作。

### 5.4 当前状态

项目处于开发阶段，`main.c` 中显示、WiFi、按键、激活模块的调用被注释，当前仅测试音频采集 → Opus编码 → Opus解码 → 播放 的闭环回路。`encoder_decoder_test_task` 作为 WebSocket 的临时替代，将编码输出直接搬运到解码输入。
